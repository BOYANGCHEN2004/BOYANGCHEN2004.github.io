# CPU実験振り返り

レポートとソースコード一式は[このレポジトリ](https://github.com/BOYANGCHENutokyo/FPU_MEMORY.git)にあります。
同じ班のコア係のCPU実験振り返りブログは[ここ](https://sqrt10.net/blogs/wdnnv10zsr4xgt8orej864qf)です。合わせて読むと、全体の流れがよりわかりやすいかもしれません。

## 役割分担

先生から発表されるグループは、たぶんハードウェア側をやりたい二人とソフトウェア側をやりたい二人で構成されていることが多いです。各係は成績をつける都合上、明確に決める必要がありますが、他の係の仕事を手伝うこと自体は禁止されていないので、第一希望が被っても話し合い次第で共同開発はできると思います。ただし、各係には単位取得要件があるので、そこだけはいかなる場合でも満たさないといけません。

## FPU・メモリ係の作業時系列

- 10,11,12月：キャッシュとマルチサイクルFPUを一通り作成 $\rightarrow$ ただ、バグを大量に埋め込んでしまったうえにデバッグもサボっていたので、実質ほとんど使いものにならず。
- 1月：諸々のテスト勉強やレポートで時間が溶け、CPU実験はいったん Stall。これは多くのグループも似たような状況だったと思います。
- 2,3月：埋め込んだバグのツケが一気に回ってきました。しかもCPU実験とは関係ないインターンプログラムで海外に行っていたため、他のグループメンバーとのコミュニケーションは完全にリモート化し、デバッグの相談相手も見つけにくくなって作業効率が低下。結局、コア係のブログでも触れられているように、完全にバグなしのメモリを提供できたのは2月下旬というありえないほど遅い時期でした。コア係さんはそれを受け取った後、一生懸命デバッグしてくれたおかげで、最終発表会までになんとか完動できました。コア係に感謝です。~~(普通に考えたらこんなに頼りないメモリ係はコア係に殺されてもおかしくない)~~ 


## 結果

130秒で2位でした。レポートでも書いたように、完全インオーダで、メモリもFPUもブロッキングだったので、ここまでの成績が出るとは正直思っていませんでした（うちの代は全体的に遅めだった、という事情もありますが）。ただ、おそらくFPUとメモリの実装が単純すぎたので、プロセッサ実験の点数面ではそこまで高く評価されなかった。同じグループ内に優上を取れたメンバーもいましたが、自分は取れませんでした。優上を目指すFPU・メモリ係は、コア側と協力してOoOまで踏み込むといいかもしれません。

## 個人的に思うFPUメモリ係として注意しておくべきこと

- 使うレジスタ量はできるだけ抑えた方がいいです。レジスタが増えすぎると implementation が通らなくなります。自分は恥ずかしながらキャッシュのタグを全部レジスタで管理しようとして、$512 \text{個} \times 14 \text{bit}$ のレジスタを使った結果、implementation が12時間経っても終わらない、という事態を引き起こしました（高前田先生曰く、これは典型的な間違いらしいです）。タグ等は BRAM で管理した方がいいです。
- キャッシュのサイズをできるだけ大きくすべき。エントリ数、ウェイ数を増やすだけで劇的に早くなる場合が多い。
- Verilator はかなり強力なです。FPU のテストは爆速(
  ?)になりますし、ビット幅の不一致みたいな凡ミスも教えてくれるので、早めに導入した方がいいです。自分はコア係さんに使い方を教えてもらってから、かなり開発効率が上がりました。
- FPU のデバッグは、仕組みを完全に理解できていなくても、最悪 LLM に投げればなんとかなる場面があります。ただ、メモリ、特に MIG IP コア周りは別で、各信号線の機能を自分で理解しておいた方がいいです。これも自分の実体験ですが、MIG IP コア周りのデバッグを LLM に投げると、AI が次々と嘘をついて症状が余計に悪化することがあります。もちろん、AI に頼らず必要なことを全部把握して、自分でデバッグするのが一番いいです。
- コア係は、メモリなしでデバッグできないことが多いです。うちのコア係さんもバグなしのメモリ・FPU を受け取った後、1週間ほどデバッグしてようやく完動できました。なので、単純な実装でもいいから、できるだけ早い段階でコアから要求されるインターフェイスを満たすメモリを作って接続すべきです。実際、接続段階ではコア係とメモリ係が一緒に波形を見ながらデバッグする必要がある場面が出てくると思います。「早期」とは、みんながテストなどで忙しくなる1月より前、つまり年内を目標にすると良い。
- 余裕があれば、コア側の実装もある程度理解しておくと良い。アーキテクチャの勉強になりますし、コアがバグったときの相談相手にもなれます（ソフト側の二人に Verilog の詳しさをあまり期待できないことも多いので）。

## それ以外に伝えておきたいこと

以下は全部、個人的なアドバイスです。他の人のブログなども見ながら、適宜参考にしてください。

- 最初から高級な設計を狙いすぎず、v1 はまず完動を第一目標にして、単純な設計で完動を目指すべきだと思います。分岐予測器、メモリのセットアソシエーティブ化、さらには動的スケジューリング、いわゆる OoO などは、v2 で一つずつ足していく方がいい気がします。早い段階で完動して単位を確保できると精神的な余裕が生まれるし、いきなり複雑な実装で完動を目指すと、デバッグが大変すぎて途中でやる気が切れ、進捗を産めなくなる可能性があります（実力とやる気が十二分にある人はこれを無視して良い）。
- 春休みに入ると、筆者のように CPU 実験と関係ない活動に時間を取られる人が出てきます。春休みの集中作業を当てにしすぎず、3A セメスターの年内完動を目標にした方が良い。
- グループメンバーのやる気や、CPU 実験に投入できる時間はそれぞれ違います。その違いが原因で人間関係が険悪になることもありえます。自分は優しいコア係さんの寛大さによって人間関係の崩壊を回避できましたが、グループ内のコミュニケーションは本当に大事です。


## 感想

- FPU・メモリのパイプライン化や、回路全体の OoO までやりたかったのですが、完動が遅すぎてそこまではできませんでした（その元凶も自分ですが...）。コア側もやりたかったようなので、時間の関係でそこまで行けなかったのは個人的にかなり悔しいですし、コア係にも申し訳なかったです。
- 前半の 10,11,12月があまりにもダラダラしていたことはかなり後悔しています。デバッグがだるい、講義で忙しい、他のメンバーもあまり進んでいないから自分も少し休みたい……など言い訳はいくらでもありますが、作った借金は結局ぜんぶ自分で返さないといけません（$\leftarrow$ CPU実験で得られた人生の教訓）。
- AI の正確性は、この数か月の CPU 実験の間にもかなり上がったと実感しています。遠くない将来、プロンプト1つで AI が CPU 実験を完結できる時代も来るのかもしれません。
